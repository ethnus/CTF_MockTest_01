Ethnus AWS Mock Test Project

Introduction

This challenge places you in a simulated AWS environment designed to test your ability to implement, secure, and troubleshoot a serverless architecture under realistic conditions.

  • Operate within the constraints of AWS Academy Learner Lab (limited permissions, restricted IAM role).
  • Diagnose and resolve misconfigurations across compute, storage, network, and access control.
  • Apply AWS best practices and critical thinking to identify gaps.
  • Develop skills in troubleshooting and architecting resilient cloud-native systems.
Important:
  • Do not inspect the bootstrap (init.sh) or evaluation (eval.sh) scripts.
  • Focus on discovery, reasoning, and resolution rather than shortcuts.
  • The objective is to build the mindset of a cloud architect and problem solver.

Resources and Purpose

Resource Purpose
S3 Bucket* Holds application data; requires encryption and restricted access.
DynamoDB Table* Stores key-value records; accessible only via private endpoints.
Lambda (Writer)* Processes incoming data, updates DynamoDB, stores objects in S3, triggers SNS.
Lambda (Reader)* Fetches and returns data via API Gateway; final output includes the success flag.
API Gateway* Private REST API exposing read operations, restricted to VPCE.
SNS Topic* Handles internal notifications for asynchronous tasks.
VPC A & B Separate networks for application components and peer services.
KMS Key* Encrypts stored data and Lambda environment variables.
EventBridge Rule* Ensures periodic activity and health verification.

Milestones (Focus Areas)

# Milestone Description
1 Secure Object Storage - Confirm encryption at rest with KMS.
- Validate strict bucket policies.
- Ensure mandatory tags are applied.
2 Data Layer Hardening - Assess DynamoDB endpoint configuration.
- Review IAM permissions for minimal access.
- Apply tagging for ownership.
3 Compute Configuration - Review Lambda memory, timeout, and concurrency.
- Ensure environment variables reference correct resources.
- Restrict permissions to only what is needed.
4 Event-driven Design - Check SNS publish permissions.
- Ensure event-driven flows are functional.
- Integrate internal alerts properly.
5 Network Security - Verify gateway and interface endpoints.
- Confirm proper VPC peering and route table configuration.
- Disallow public exposure.
6 API Exposure Control - Ensure API Gateway is PRIVATE.
- Validate policy enforces VPCE-only access.
- Block unauthorized access routes.
7 Monitoring & Scheduling - Ensure EventBridge rule is enabled.
- Confirm regular execution and observability.
- Integrate with operational monitoring.
8 Encryption Management - Review KMS grants for least privilege.
- Validate integration with S3, DynamoDB, and Lambda.
- Plan for key lifecycle management.
9 Integration Testing - Run full end-to-end data flow tests.
- Validate outputs and side effects.
- Document troubleshooting steps.
10 Resilience & Tagging - Confirm consistent tagging across all resources.
- Ensure architecture supports operational resilience.
- Align with cost and compliance requirements.

Best Practices

  • Follow the AWS Well-Architected Framework for Security, Reliability, and Operational Excellence.
  • Use least privilege everywhere—restrict IAM and resource policies tightly.
  • Encrypt data at rest and in transit; ensure KMS integration throughout.
  • Tag all resources consistently for accountability and cost management.
  • Monitor, log, and verify each change you make—don’t assume correctness.
  • Use CLI and CloudWatch logs to validate real behavior, not assumptions.
  • Document findings and improvements as part of your process.

Completing this project demonstrates proficiency in building, securing, and troubleshooting cloud-native applications within AWS under real-world constraints.
